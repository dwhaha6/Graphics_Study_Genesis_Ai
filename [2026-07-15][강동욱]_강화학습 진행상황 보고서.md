
# 강화 학습 세팅 정리

### 기본 물리/시간 세팅(Hz와 ms는 역수 관계)

$$  
\mathrm{ms} = \frac{1000}{\mathrm{Hz}}  
$$

| 항목                      | 값                                 |
| ----------------------- | --------------------------------- |
| **DT**                  | **25ms (40Hz)** -> 0.025 x 40 = 1 |
| Substeps                | 10 (물리계산은 2.5ms)                  |
| 제어 주기 (follower/turret) | 100ms (CONTROL_EVERY=4)           |
| Bot 결정 주기               | 1Hz                               |
| 매치 길이                   | 60 sim초 (2,400 physics step)      |
- 제어주기에서 CONTROL_EVERY = 4 라는 건 
```
Step 0   → 조향 계산
Step 1   → 이전 조향 유지
Step 2   → 이전 조향 유지
Step 3   → 이전 조향 유지
Step 4   → 다시 계산
```
- 포탑 회전이나 조향 같은 저수준 제어는 4step마다 업데이트된다는 것
	- 이때 저수준이라는 건 전략적 사고나 판단과 비교하여 저수준이라는 뜻
- 반면 Bot AI가 결정을 내리는 고수준 제어는 1초(40step)마다 한 번 실행됨

### 4번 사항 설명

- GPU 계산 완료 대기, CPU 복사를 18번 반복하는 구조
```
위치 읽기 → sync 
속도 읽기 → sync 
포탑 상태 읽기 → sync 
... 총 18회
```
#### after
- 18번 기다리던 것을 1번만 기다리게 됨
```
physics step 완료 
↓ 
필요한 state를 한꺼번에 GPU→CPU로 복사 
↓ 
그 복사본을 combat, turret, observation 계산에서 재사용
```

### 5번 사항 설명

```python
for env in range(n_envs):
    for tank in range(n_tanks):
        combat 계산
        camo 계산

```
- 이렇게 Python 레벨 반복문을 768번 돌게 하지 말고
#### after

```python
shape = (n_envs, n_tanks, features)

```

## 그림 설명
### 1. 관측 벡터 입력
- 위에 나온 x,y,hp 등 scalar 1개 = 토큰 1개
- 각 scalar들이 벡터 형태로 들어오면 scalar 하나하나가 Linear embedding을 거침
	- x(1) --Linear--> 256차원
	- y(1) --Linear--> 256차원
### 2. Linear embedding
- Transformer 의 token embedding에 해당
- 앞서 말한 토큰을 논문 기준 256차원으로 펼쳐줌
- wx+b(w,b는 학습 대상)
### 3. Positional embedding
- Transformer에서는 positional이 문장 각 토큰의 순서 정보를 의미했음
- 여기선 attribute(이 값이 어떤 팀, 어떤 agent의 어떤 속성인지 hp인지 위치 정보인지 등)을 의미
- 순서 정보는 의미가 없기에 따로 부여되지 않음(100이라는 값이 hp라는 사실만 알면 ok)
	- 예를 들어 agent의 1번 아군 tank의 현재 x위치 정보 토큰이 있다면
		- 이건 x좌표라는 종류의 정보를 담은 attribute id vector
		- 이건 1번 tank라는 정보를 담은 agent index vector
		- 아군이라는 정보를 담은 team vector
		- 현재 time step이라는 정보를 담은 timestep vector(RNN구조 없이 과거 정보와 현재 정보 구분 가능)
### 4. Transformer encoder
- 여긴 기존 인코더 구조와 동일
- dimension = 256, head = 8의 인코더 블록 8개
- 각 토큰들은 attention 과정을 통해 다른 토큰들의 문맥과 섞이게 되고 결과적으로 encoder의 출력은 **obs를 요약한 것 뿐만 아니라 모델이 그 obs를 기반으로 어떻게 행동하고자 한다라는 정보**도 담겨져 있는 것
### 5. Actor/Critic head
- Actor/Critic 각각 encoder block 1개 + LayerNorm + Linear(256 -> n) 구조를 가짐
	- actor는 n=20, critic은 n=400
- 논문에 제시된 전체 파라미터는 7,232,256 (7.23M)

# 현재 tank 강화 학습에 맞게 모델 재구성
- 학습된 transformer encdoer까지는 그대로 두고 actor와 critic을 재구성
- Hugging face에 제공된 모델의 actor는 StarCraft 행동 20개에 대해 정의되어 있음 -> actor 교체 이유
- 제공된 critic은 여러 개의 행동들 각각을 평가하여 best가 무엇인 지를 고르는 역할을 수행했다면 PPO에 필요한 critic은 행동과 별개로 현재 주어진 상황이 어떤 지를 평가하는 것이기에 critic 또한 새로 교체
- actor, critic은 우리의 것으로 갈아끼우고 MARL GPT의 **전투 이해 능력**을 빌리자는 것

![](./이미지/Revised_MARL_GPT.png)

| 역할     | 원본 구현                                         | 우리 구현                            |
| ------ | --------------------------------------------- | -------------------------------- |
| Actor  | block+linear → StarCraft 행동 20개               | **move/aim/fire head** ← 새 actor |
| Critic | block+linear → Q(o,a) 분포 (20 action × 20 bin) | **value head** V(o) ← 새 critic   |
- 추가로 원본에선 actor critic 각각에 encoder block을 두었으나, 여기선 encoder 출력 직후 256차원의 벡터 87개를 평균내어 256차원 벡터 1개를 만들고 이 벡터를 256 -> n 구조의 head에 입력시킴
	- 현재 우리 학습 환경에서의 명령이 단순(이동, 공격 대상 지정)하므로 그 구조도 단순하게 변화시킴
	- 이로 인해 아래와 같이 파라미터 개수가 줄어듦(학습 시간 관점에서는 이득)

### 원본 대비 파라미터 변화(7M->5M)

|                                      | 파라미터                  | 비고     |
| ------------------------------------ | --------------------- | ------ |
| **원본 MARL-GPT 전체**                   | **7,232,256 (7.23M)** |        |
| ├ Embedding 5종                       | 41,728                | 유지     |
| ├ 공유 encoder block × 8               | 5,508,608             | 유지     |
| ├ Actor head (block+LN+Linear)       | 792,320               | **제거** |
| └ Critic head (block+LN+Linear)      | 889,600               | **제거** |
| **우리 TankMGPT 전체**                   | **5,553,163 (5.55M)** |        |
| ├ 이식 (embedding + encoder)           | 5,550,336             | 원본 그대로 |
| └ 신규 head (LN + linear 4개 + log_std) | **2,827**             | 새로 학습  |


### 임의로 추가한 actor/critic 설명

| Head      | 구조                  | 출력                        | 설명                                                                                                                                  |
| --------- | ------------------- | ------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **move**  | Linear 256→2 + tanh | 목적지 (x, y) 의 Gaussian 분포  | 이동 목적지 (x,y)를 출력함, 출력은 [-1,1]로 정규화 된 맵 좌표이며 가우시안 분포를 활용해 이동. 예를 들어 (20,15)로 이동명령이 내려지면 그 점을 평균으로 하고 σ를 분산으로 한 정규 분포에서 목적지를 선택       |
| **aim**   | Linear 256→4        | Categorical (없음/적1/적2/적3) | 조준 이전에 사격 불가한 경우(생존 여부, 위장 상태, 시야 차단 등)을 검증 후 조준 → turret 이 그 target 자동 추적                                                          |
| **fire**  | Linear 256→2        | 쏜다/안 쏜다                   | 현재는 조준한 적을 사격 가능할 시 자동으로 무조건 발사하는 구조, 향후 더욱 고도의 전술(위장 유지를 위해 발사X)이 가능해지면 활용할 계획                                                     |
| **value** | Linear 256→1        | V(s) 스칼라                  | critic 역할, 시작부터 랜덤 파라미터의 critic이 개입하면 actor도 망가질 수 있으므로 0~4 iter까지는 actor가 알아서 하게끔 두고 critic은 학습 그 이후부터는 critic이 개입하며 actor의 행동을 조절 |
- 이동 방식은 서로 같은 모델 구조이더라도 분포에서 목적지를 랜덤하게 고르므로 각 tank마다 노이즈가 낀 다른 이동을 하게 됨 -> 이동이 달라지니 obs가 달라지고 aim등의 다른 판단도 달라짐 -> 더 다양한 행동양상 도출 가능
- Actor에 해당하는 3개 head는 rule bot 기반으로 지도학습된 head로 start(랜덤 파라미터가 아님)
- 헷갈릴 수 있는 point로 이 head들은 행동 결정을 내리는 것이 아니라 encoder에서 내린 함축된 벡터를 잘 해석해서 우리가 원하는 차원의 실제 명령으로 mapping시켜주는 것
